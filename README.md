# Synthetic Data Augmentation for YOLO Object Detection

This university Computer Vision project measures the synthetic-to-real domain gap in YOLO object
detection. It will compare models trained with controlled proportions of real and synthetic images,
then evaluate every final model on the same fixed, held-out real test set.

## Research objective

The central question is how training-data composition affects performance on real images. The
pipeline is designed so that changes in results can be attributed to the real/synthetic mix rather
than an inconsistent test set or an uncontrolled training budget.

## Planned architecture

- `src/synthdet`: reusable configuration, data, synthetic generation, training, evaluation, and
  inference packages.
- `scripts`: thin, reproducible command-line entry points.
- `configs`: version-controlled project and, later, experiment configurations.
- `apps/api`: future FastAPI inference service.
- `apps/web`: future Next.js dashboard.
- `datasets`, `artifacts`, and `models`: local-only large inputs and generated outputs.
- `reports`: version-controlled narrative with generated figures and tables excluded by default.
- `docs`: scientific protocol and decision history.

## Primary experiments

| Regime | Real share | Synthetic share |
| --- | ---: | ---: |
| `synthetic_only` | 0% | 100% |
| `real_25` | 25% | 75% |
| `real_50` | 50% | 50% |
| `real_75` | 75% | 25% |
| `real_only` | 100% | 0% |

Primary regimes should use an equal total training-image budget whenever dataset availability
permits. An optional `best_mixed_domain_randomized` ablation will compare the best mixed regime
against the same mixture with stronger domain randomization.

## Non-negotiable evaluation rule

The real test split is created once, recorded, and kept unchanged across every experiment. Test
images may never be used as training or validation samples, copy-paste sources, or synthetic
backgrounds. Results must be generated by the evaluation pipeline and traceable to an experiment
configuration and output directory; metrics must never be entered manually or fabricated.

## Installation

Python 3.11 or newer is required. From the repository root:

```bash
python -m venv .venv
```

Activate the environment on Windows PowerShell:

```powershell
.venv\Scripts\Activate.ps1
```

Or on Linux/macOS:

```bash
source .venv/bin/activate
```

Install the project and development tools, then validate the foundation:

```bash
python -m pip install -e ".[dev]"
python scripts/check_environment.py
pytest
ruff check .
```

Copy `.env.example` to `.env` only when local overrides are needed. Do not commit `.env`.
